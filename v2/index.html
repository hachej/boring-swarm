Pull-Based Queues Durable State SQLite Backend YAML DSL

Core Concepts

Beads

Work items with labels, comments, assignee. Source of workflow truth stored in SQLite.

Queues

Defined by source label (e.g., needs-impl). Workers autonomously pull unassigned beads.

Workers

Processes that: claim → work → emit STATE → release. Workers are ephemeral.

bsw

Dumb executor: parse DSL, claim beads atomically, validate STATE events, transition labels.

Orchestrator

Smart controller (user-built): runs queues, reads status/logs, sends mail, decides retries.

State

Determined by bead labels only. No hidden state. Restart-safe by design.

How Code Is Run (Execution Flow)

When orchestrator calls bsw run flows/queue.yaml, here's what happens step-by-step.

Command Line

bsw run flows/implement_queue.yaml --mode oneshot # Blocks until queue is empty, then exits with status

Internal Execution (6 Steps)

STEP 1
Parse & Validate

Load YAML. Validate schema. Extract source label, worker count, transitions.

STEP 2
Claim Bead

Query beads in source label. Find unassigned. Atomically set assignee + token via SQLite CAS. Emit spawn event.

STEP 3
Spawn Worker

Fork process with bead context. Pass assignment_token. Start state detectors (JSONL watcher, process monitor, stuck timer).

STEP 4
Worker Works

Worker: get bead → run prompt → emit STATE event with token. bsw watches for comments.

STEP 5
Detect State

3-layer: JSONL events, process exit, stuck timer. Validate STATE against transitions. Emit state event.

STEP 6
Atomic Transition

Remove source label, add target, clear assignee. Emit transition event. Release worker.

Loop: Repeat steps 2-6 until source label is empty AND all workers are idle.
Exit: oneshot mode: SIGTERM all workers. service mode: keep running for new beads.

State Transitions

Defined in DSL. Example:

transitions: "impl:done": "needs-proof" # Done implementing → move to proof queue "impl:blocked": "needs-help" # Blocked → escalate to help queue

How Status Is Extracted (3 Mechanisms)

Orchestrator never asks bsw "what's the state?" Instead, it observes via three independent channels.

Mechanism 1: JSONL Event Stream (Real-Time Log)

File: .bsw/logs/queue-<run-id>.jsonl

Ordered sequence of events. Orchestrator tails this file asynchronously (no blocking).

{"event":"spawn","bead_id":"bd-123","token":"run:bd-123:1","pid":4521,"ts":"2026-02-27T10:00:00Z"} {"event":"state","bead_id":"bd-123","value":"impl:done","token":"run:bd-123:1","ts":"2026-02-27T10:05:00Z"} {"event":"transition","bead_id":"bd-123","from":"needs-impl","to":"needs-proof","ts":"2026-02-27T10:05:01Z"} {"event":"done","bead_id":"bd-123","ts":"2026-02-27T10:05:01Z"}

Mechanism 2: Status Snapshot (One-Call Query)

Command: bsw status --json

Point-in-time snapshot of all queue state, bead ownership, and attention items.

{ "queues": { "needs-impl": {"total": 7, "unassigned": 3, "assigned": 4} }, "beads": [ { "bead_id": "bd-123", "label": "needs-impl", "assignee": "GreenLake", "activity_state": "active", "last_progress": "2026-02-27T10:12:03Z" } ], "agents": [ { "agent_name": "GreenLake", "provider": "codex", "session_ref": "thread_abc123", "resume_command": "codex app-server --thread thread_abc123" } ], "attention": [ { "bead_id": "bd-123", "reason": "waiting_input", "suggested_action": "send_mail_then_ping" } ] }

Mechanism 3: Provider State Detection (Auto-Detect)

bsw automatically detects worker state from three independent signals:

Signal Source Detects
Provider Events Claude/Codex JSON output in comments Explicit STATE events with assignment token
Process Exit OS process table monitoring Worker crash (SIGKILL, OOM, panic, non-zero exit)
Stuck Timer Elapsed time since last progress Hung worker (paused if state=waiting_input)

Orchestrator Usage Pattern

  1. Start: bsw run flows/queue.yaml --mode oneshot (blocking call)
  2. Monitor: Tail .bsw/logs/queue-*.jsonl asynchronously in real-time
  3. Check: bsw status --json whenever you need a full snapshot
  4. Act: Use attention list to know what needs intervention

Queue DSL (YAML Specification)

Define queues in YAML. Queue = source label + worker count + state transitions.

Complete Example

version: "1" name: "implement_queue" source: label: "needs-impl" workers: count: 3 timeout: 3600 transitions: "impl:done": "needs-proof" "impl:blocked": "needs-help" "impl:invalid": "error"

Field Reference

Field Type Description
version string DSL version (currently "1")
name string Queue identifier (used in logs)
source.label string Label defining queue (beads with this label are work items)
workers.count int Max concurrent workers (backpressure valve)
workers.timeout int (seconds) How long before worker considered stuck (paused if waiting_input)
transitions map STATE event → target label. Keys are event names, values are destination labels

Execution Models

Best for: Strict ordering. Queue 1 completes before Queue 2 starts.

bsw run flows/implement.yaml # Blocks until done bsw run flows/proof.yaml # Blocks until done bsw run flows/review.yaml # Blocks until done

Clear exit codes and process semantics. Orchestrator can check each return code.

Best for: Multiple independent queues run in parallel with wait.

bsw run flows/impl.yaml --mode service & bsw run flows/proof.yaml --mode service & bsw run flows/review.yaml --mode service & wait

Each queue refills as upstream produces work. Orchestrator waits for all to complete.

Best for: Long-running queue that continuously drains refilled work.

bsw run flows/review.yaml --mode service

Stays active until orchestrator calls bsw stop or process killed.

Key Contracts

Worker STATE Contract

Worker emits one terminal STATE comment with assignment token:

STATE impl:done assignment=run:bd-123:1

bsw validates: (1) event exists in transitions, (2) token matches active claim, (3) comment id newer than cursor.

Claim Contract (Atomic)

  • bsw claims via CAS: only succeeds if bead currently unassigned
  • If claim fails, emit claim_conflict and try next candidate
  • Worker must never start without successful claim
  • Release on timeout/crash must be guarded: clear assignee only if it matches active worker

Transition Contract (Atomic)

  • Remove source label
  • Add target label from transitions[event]
  • Clear assignee
  • If precondition fails (stale STATE, wrong token, missing target label): emit transition_conflict

Stuck Contract

  • If no progress past timeout AND state ≠ waiting_input: emit stuck_no_progress
  • If state = waiting_input: pause stuck escalation timer
  • No automatic retry/escalation inside bsw

Idempotency Contract

  • Persist last_processed_comment_id per bead
  • Only process comments with id > cursor
  • Ignore stale/duplicate STATE comments

Attention to Action Mapping

When bsw status --json returns attention items, orchestrator uses this table to know what to do:

Attention Reason Meaning Operator Action
waiting_input Agent requires approval/feedback Send mail response, then bsw ping <bead-id>
stuck_no_progress No progress and not waiting_input Inspect resume_command; optionally bsw kill <bead-id> and rerun
orphaned_assignment Registry/process/assignee mismatch Reconcile and clear stale claim, then requeue
transition_conflict Preconditions failed during transition Refresh bead state, apply manual repair if needed

Design Principles

  1. Beads are workflow truth: Labels, comments, assignee define durable state (not internal bsw state)
  2. Single owner for mutations: Only bsw can assign or move labels. Workers emit STATE, nothing else.
  3. Pull-based queue: Workers pull when ready. No push complexity. Natural backpressure.
  4. bsw is dumb: Queue executor only. Orchestrator owns policy (retry, escalation, approval).
  5. Structured IO only: JSONL events, JSON status. No regex parsing of text.
  6. Operator-first visibility: One status call exposes everything: queue state, ownership, attention.
  7. Restart-safe by design: Session UUIDs, idempotent STATE handling. Crash → rerun queue safely.
